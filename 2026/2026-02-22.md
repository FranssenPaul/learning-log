## 1. Les images Docker — anatomie

Une image c'est un **système de fichiers figé en couches**. Elle ne tourne pas, elle ne consomme pas de CPU. C'est une recette reproductible.

```
Image hello-docker
├── Debian slim       ← hérité de FROM python:3.12-slim
├── Python 3.12       ← hérité de FROM python:3.12-slim
└── main.py           ← copié par COPY
```

**Ton code est du texte** — `main.py` n'est pas un ELF, c'est un fichier ASCII que Python (lui un ELF) interprète :

```bash
file main.py
# Python script, ASCII text   ← juste du texte
```

**Le Dockerfile complet :**
```dockerfile
FROM python:3.12-slim    # image de base (Debian + Python préinstallé)
WORKDIR /app             # dossier de travail dans le conteneur
COPY . .                 # TOUT le code copié dans l'image
CMD ["python", "main.py"] # commande au démarrage = PID 1
```

**`COPY . .` — deux points :**
- Premier `.` → tout le dossier courant sur ta machine
- Deuxième `.` → dans le WORKDIR du conteneur (`/app`)

## 2. Les conteneurs — cycle de vie

```
Dockerfile → docker build → image → docker run → conteneur
```

**Le conteneur = ton app qui tourne** — un processus isolé avec tout ce dont il a besoin.

```
Conteneur hello-docker
└── python main.py (PID 1) → print → exit → conteneur mort

Conteneur PostgreSQL
└── postgres (PID 1) → écoute port 5432 → reste en vie indéfiniment
```

**La règle :** le conteneur vit exactement le temps que vit son PID 1.

**Conteneurs éphémères — très utiles :**
```bash
docker run --rm mon-app alembic upgrade head  # migrations
docker run --rm pandoc doc.md doc.pdf         # conversion
docker run --rm pytest                         # tests CI/CD
```

## 3. Les variables d'environnement

La façon standard de configurer un conteneur sans modifier le code :

```python
import os
nom = os.getenv("NOM", "Docker")  # valeur par défaut = "Docker"
print(f"Hello {nom} !")
```

```bash
docker run -e NOM="Paul" hello-docker
# affiche : Hello Paul !
```

**Pourquoi c'est fondamental :**
```bash
# dev
docker run -e DATABASE_URL="localhost:5432" mon-app

# production Cotonou
docker run -e DATABASE_URL="192.168.1.100:5432" mon-app
```

Même image, comportements différents selon l'environnement.

## 4. UV + Docker — la combinaison idéale

UV gère les dépendances proprement avec `uv.lock` — Docker utilise ça pour des builds reproductibles :

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY pyproject.toml uv.lock .    ← en premier pour profiter du cache
RUN pip install uv && uv sync --frozen  ← exactement ce qui est dans le lock
COPY . .                          ← le code en dernier
CMD ["uv", "run", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**`--frozen`** = installe exactement ce qui est dans le lock, rien d'autre.

**Cache Docker intelligent :**
```
COPY pyproject.toml uv.lock .  ← couche cachée si pas changé
RUN uv sync --frozen            ← couche cachée si pas changé
COPY . .                        ← rebuild seulement ici si code change
```

**`--host 0.0.0.0`** — obligatoire en Docker, sinon FastAPI n'est accessible que depuis l'intérieur du conteneur.

## 5. Les namespaces — comprendre l'isolation

**Un namespace = une vue isolée d'UNE ressource du noyau.**

```
Types de namespaces :
├── pid   → vue isolée des processus
├── net   → vue isolée du réseau (interfaces, ports)
├── mnt   → vue isolée du filesystem
├── uts   → vue isolée du hostname
├── user  → vue isolée des utilisateurs
├── ipc   → vue isolée de la communication inter-processus
├── cgroup → vue isolée des groupes de ressources
└── time  → vue isolée de l'horloge
```

**Le concept clé :**
```
PID = concept (numéroter les processus), inventé une fois dans le noyau

Namespace système   → PID 1 = systemd
Namespace conteneur → PID 1 = postgres     ← même noyau, vue différente
Namespace Chrome    → PID 1 = renderer     ← Chrome fait pareil !
```

**Analogie :** les numéros de rue — le concept existe une fois, chaque rue a sa propre numérotation sans confusion.

**Créer un namespace manuellement :**
```bash
sudo unshare --pid --fork --mount-proc bash
echo $$    # → 1  (tu es PID 1 !)
ps aux     # → seulement bash et ps, le monde réel a disparu
exit
```

Docker c'est essentiellement `unshare` + image filesystem + cgroups + CLI sympa.

**Ce que tu as vu sur ta machine :**
```bash
lsns
# Chrome isole chaque onglet dans son propre namespace PID !
# Snap a ses propres namespaces mnt
# Ton PostgreSQL Docker a ses propres namespaces
```

## 6. Les cgroups — limiter les ressources

Les namespaces isolent la **vue**, les cgroups limitent les **ressources** :

```
cgroups contrôle :
├── memory  → RAM maximale
├── cpu     → % de CPU autorisé
├── io      → vitesse lecture/écriture disque
├── cpuset  → quels cœurs CPU autorisés
└── pids    → nombre maximum de processus
```

**Ton PostgreSQL sans limites :**
```bash
docker stats thinkplay-postgres-1
# RAM : 45 Mo / 5.7 Go  ← limit = max (toute ta RAM)
# CPU : 0.00%           ← au repos
# PIDs : 6              ← 6 processus internes postgres
```

**En production pour ton POS :**
```yaml
services:
  db:
    mem_limit: 512m    ← PostgreSQL limité
  api:
    mem_limit: 256m    ← FastAPI limité
    cpus: "0.5"        ← max 50% CPU
```

Sans limites, un conteneur sous charge peut affamer les autres — le restaurant ne peut plus encaisser.

**La vision complète Docker :**
```
Namespaces  → isolation (qui voit quoi)
Cgroups     → limitation (combien de ressources)
Filesystem  → l'image (quoi comme système)
```

## 7. Le réseau Docker

```bash
docker network ls
# bridge              ← réseau par défaut
# host                ← partage le réseau de l'hôte (pas de namespace NET)
# none                ← isolation totale, pas de réseau
# thinkplay_default   ← créé automatiquement par docker-compose
```

**Chaque conteneur a sa propre IP :**
```
Réseau thinkplay_default (172.18.0.0/16)
├── gateway   172.18.0.1  ← porte vers Ubuntu
└── postgres  172.18.0.2  ← ton PostgreSQL
```

**Docker fait le DNS interne :**
```python
# dans ton FastAPI dockerisé
DATABASE_URL = "postgresql://user:pass@postgres:5432/db"
#                                        ↑
#                              nom du service → Docker résout en 172.18.0.2
```

**La sécurité par défaut :**
```
Internet
  ↓ port 8000 exposé
conteneur FastAPI (172.18.0.3)
  ↓ réseau interne Docker
conteneur PostgreSQL (172.18.0.2)  ← jamais exposé à internet !
```

## 8. Les volumes — persistance des données

**Sans volume :** les données meurent avec le conteneur.
**Avec volume :** les données survivent à tout.

```bash
docker volume inspect thinkplay_pgdata
# Mountpoint: /var/lib/docker/volumes/thinkplay_pgdata/_data
```

Tes données PostgreSQL vivent physiquement sur ton Ubuntu — pas dans le conteneur.

```
Conteneur PostgreSQL
└── /var/lib/postgresql/data/  ← vue depuis le conteneur
      ↕ bind mount
Ton Ubuntu
└── /var/lib/docker/volumes/thinkplay_pgdata/_data/  ← réalité physique
```

**Quand tu éteins le PC :**
```
PC éteint → conteneur mort → dossier Ubuntu intact
PC rallumé → nouveau conteneur → retrouve ses données
           → "Skipping initialization" dans les logs ✅
```

**La règle absolue :**
```
Volume nécessaire      → données PostgreSQL, fichiers uploadés, logs
Pas de volume          → code FastAPI (dans l'image), dépendances
```

**Pour thinkplay — tes fichiers markdown :**
```yaml
services:
  api:
    volumes:
      - ./questions:/app/questions  ← bind mount, éditable via VS Code SSH
```

## 9. Architecture complète du POS

```
Serveur HP EliteDesk Cotonou
└── Docker
      ├── conteneur Nginx        ← port 80, sert React + reverse proxy
      ├── conteneur FastAPI      ← port 8000, ton code Python
      └── conteneur PostgreSQL   ← port 5432 interne seulement
            └── volume postgres_data (données persistantes)

Tablettes du restaurant
└── navigateur Chrome/Safari
      └── React (téléchargé depuis Nginx, tourne dans le navigateur)
```

**Le flux complet :**
```
Tablette → requête HTTP → Nginx
Nginx → /api/... → FastAPI (réseau Docker interne)
FastAPI → SQL → PostgreSQL (réseau Docker interne)
PostgreSQL → données → FastAPI → Nginx → tablette
```

**docker-compose.yml final :**
```yaml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    restart: always

  api:
    build: .
    environment:
      - DATABASE_URL=postgresql://user:pass@db:5432/pos
    depends_on:
      - db
    restart: always

  db:
    image: postgres:16-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: always

volumes:
  postgres_data:
```

**`restart: always`** — indispensable pour Cotonou : coupure de courant, redémarrage serveur → tout repart automatiquement.

## 10. SSH et infrastructure distante

**SSH server = un daemon** qui écoute sur le port 22 et attend des connexions :

```
sshd      → écoute port 22, attend des connexions distantes
postgres  → écoute port 5432, attend des requêtes SQL
nginx     → écoute port 80, attend des requêtes HTTP
```

Même philosophie — des programmes qui tournent en permanence et attendent.

**Authentification :**
```bash
# générer une paire de clés
ssh-keygen -t ed25519

# clé privée → reste sur ton laptop (secret absolu)
# clé publique → copiée sur le serveur

ssh-copy-id paul@192.168.1.100
ssh paul@192.168.1.100  ← connexion sans mot de passe
```

**VS Code Remote SSH :**
```
Ton laptop Ubuntu
└── VS Code (juste un écran)
      ↓ SSH
Serveur (VM, VPS, HP EliteDesk)
      └── ton code, Docker, les données
```

Le laptop n'est qu'un éditeur de texte — toute la puissance est sur le serveur.

## 12. Bonne pratique — dockeriser dès le début

```
✅ Dès le jour 1 du projet → Dockerfile minimal
✅ Chaque backend → dockerisé
✅ restart: always → pour tous les services en prod
✅ volumes → pour toutes les données persistantes
✅ réseau Docker → PostgreSQL jamais exposé à internet
✅ uv.lock → reproductibilité garantie
```

**Plusieurs projets sur un serveur — zéro conflit :**
```
Serveur VPS
└── Docker
      ├── POS      : FastAPI + PostgreSQL (port 5432)
      ├── Thinkplay: FastAPI + PostgreSQL (port 5433)
      └── paul.craft: Flask + SQLite
```

Chaque projet a son propre réseau Docker, sa propre base, ses propres volumes.
