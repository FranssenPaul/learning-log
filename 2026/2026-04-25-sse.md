# Apprendre SSE — Server-Sent Events

> Objectif : comprendre **strictement la technologie SSE**, son fonctionnement réseau, son format, ses limites, sa reconnexion, puis s’entraîner avec des questions de retrieval practice.

---

## 1. Le problème que SSE résout

Dans une application web classique, le navigateur demande des données au serveur :

```txt
client  →  serveur : GET /orders
serveur →  client  : réponse JSON
```

Mais certaines interfaces ont besoin d’être averties quand quelque chose change :

- nouvelle commande ;
- changement de statut ;
- notification ;
- progression d’une tâche ;
- mise à jour d’un tableau de bord.

Sans mécanisme temps réel, on utilise souvent du polling :

```txt
Toutes les 2 secondes :
client → GET /sync
```

Cela fonctionne, mais génère des requêtes même quand rien ne change.

SSE propose une autre idée :

```txt
Le client ouvre une connexion HTTP.
Le serveur garde la réponse ouverte.
Quand il a quelque chose à dire, il écrit dans cette réponse.
```

---

## 2. Définition simple

**SSE**, ou **Server-Sent Events**, est une technologie web qui permet au serveur d’envoyer des événements au navigateur à travers une connexion HTTP persistante.

Le navigateur utilise l’API JavaScript :

```js
const source = new EventSource("/stream");

source.onmessage = (event) => {
  console.log(event.data);
};
```

Le serveur répond avec un flux de texte au format :

```http
Content-Type: text/event-stream
```

Puis il écrit des messages comme :

```txt
id: 42
event: order_updated
data: {"order_id":123}

```

La ligne vide à la fin indique :

```txt
l’événement est terminé, le navigateur peut le transmettre au JavaScript.
```

---

## 3. SSE dans la pile réseau

SSE n’est pas un protocole totalement nouveau comme WebSocket.

Il s’appuie sur HTTP.

```txt
Application : EventSource / SSE
HTTP        : GET /stream, réponse text/event-stream
TCP         : connexion fiable entre client et serveur
IP/Wi-Fi    : transport réseau local ou internet
```

### 3.1 Ce qui se passe concrètement

Le navigateur envoie une requête HTTP classique :

```http
GET /stream HTTP/1.1
Host: pos.local
Accept: text/event-stream
Cache-Control: no-cache
```

Le serveur répond :

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

Puis la réponse ne se termine pas immédiatement.

Le serveur continue d’écrire dans le flux :

```txt
id: 1
event: hello
data: connected

id: 2
event: order_updated
data: {"id":123}

```

Le point important est celui-ci :

```txt
SSE = une réponse HTTP longue, gardée ouverte.
```

---

## 4. HTTP/1.1, HTTP/2 et connexions persistantes

### 4.1 En HTTP/1.1

En HTTP/1.1, SSE occupe une connexion TCP ouverte.

Le navigateur garde cette connexion disponible pour lire les événements.

Problème : les navigateurs limitent généralement le nombre de connexions simultanées par domaine. MDN indique une limite pratique de **6 connexions SSE par navigateur + domaine** en HTTP/1.x.

Cela veut dire :

```txt
Si une même tablette ouvre plusieurs onglets vers le même serveur,
on peut atteindre la limite plus vite.
```

Dans une application POS normale, on évite de multiplier les onglets et on garde une seule connexion SSE par client.

### 4.2 En HTTP/2

Avec HTTP/2, plusieurs flux peuvent passer dans une même connexion TCP.

Cela rend SSE plus confortable, car une connexion HTTP/2 peut transporter plusieurs streams en parallèle.

MDN indique que la limite est alors négociée entre le client et le serveur, avec une valeur par défaut courante autour de 100 streams.

### 4.3 SSE ne fait pas d’upgrade de protocole

Contrairement à WebSocket, SSE ne commence pas par :

```http
Upgrade: websocket
```

SSE reste une réponse HTTP normale.

C’est une raison pour laquelle SSE traverse souvent plus facilement les proxies, reverse-proxies et infrastructures HTTP classiques.

## 5. Format exact d’un événement SSE

Un événement SSE est composé de lignes.

Les champs principaux sont :

```txt
id: 123
event: order_updated
data: {"order_id":456}
retry: 3000

```

### 5.1 `data`

`data` contient la donnée envoyée au client.

Exemple :

```txt
data: hello

```

Côté JavaScript :

```js
source.onmessage = (event) => {
  console.log(event.data); // "hello"
};
```

Si plusieurs lignes `data:` se suivent, elles sont concaténées avec des retours à la ligne.

```txt
data: première ligne
data: deuxième ligne

```

Le navigateur reçoit :

```txt
première ligne\ndeuxième ligne
```

### 5.2 `event`

`event` donne un nom à l’événement.

Exemple :

```txt
event: order_updated
data: {"id":123}

```

Côté client :

```js
source.addEventListener("order_updated", (event) => {
  const payload = JSON.parse(event.data);
  console.log(payload.id);
});
```

Sans champ `event`, le navigateur déclenche `message`.

### 5.3 `id`

`id` donne un identifiant à l’événement.

```txt
id: 184
event: stock_updated
data: {"product_id":9}

```

Le navigateur garde en mémoire le dernier `id` reçu.

Si la connexion tombe puis se réouvre, il peut envoyer automatiquement l’en-tête :

```http
Last-Event-ID: 184
```

Cela permet au serveur de savoir à partir de quel événement il doit reprendre.

### 5.4 `retry`

`retry` indique au navigateur combien de millisecondes attendre avant de tenter une reconnexion.

```txt
retry: 5000

```

Cela signifie :

```txt
Si la connexion tombe, attends environ 5 secondes avant de réessayer.
```

### 5.5 Commentaires et heartbeat

Une ligne qui commence par `:` est un commentaire.

Exemple :

```txt
: heartbeat

```

Le navigateur l’ignore côté JavaScript, mais le message garde la connexion active.

C’est utile contre :

- les proxies qui ferment les connexions inactives ;
- les routeurs qui coupent les flux silencieux ;
- les connexions zombies difficiles à détecter.

Un heartbeat typique :

```txt
: ping

```

envoyé toutes les 15 à 30 secondes.

## 6. Cycle de vie d’une connexion SSE

### 6.1 Ouverture

```js
const source = new EventSource("/stream");
```

Le navigateur :

1. envoie `GET /stream` ;
2. attend une réponse `text/event-stream` ;
3. garde la connexion ouverte ;
4. parse les événements ligne par ligne.

### 6.2 Réception

```js
source.onmessage = (event) => {
  console.log(event.data);
};
```

Ou avec un événement nommé :

```js
source.addEventListener("order_updated", (event) => {
  console.log(JSON.parse(event.data));
});
```

### 6.3 Erreur et reconnexion

```js
source.onerror = () => {
  console.log("SSE disconnected or reconnecting...");
};
```

Le navigateur tente normalement de se reconnecter automatiquement.

### 6.4 Fermeture volontaire

```js
source.close();
```

Cela coupe la connexion côté client.

---

## 7. `Last-Event-ID` et reprise après coupure

La reprise est une des grandes forces de SSE.

Supposons que le client reçoive :

```txt
id: 100
data: event 100

id: 101
data: event 101

```

Puis le Wi-Fi tombe.

Quand le navigateur reconnecte, il peut envoyer :

```http
Last-Event-ID: 101
```

Le serveur peut alors reprendre à :

```txt
102, 103, 104, ...
```

Mais attention : cela suppose que le serveur conserve un journal des événements.

SSE ne crée pas automatiquement l’historique. C’est à l’application de le faire.

Donc, pour une vraie robustesse :

```txt
SSE + event_log + endpoint /sync
```

## 8. Comparaison avec polling, long polling et WebSocket

| Technologie | Sens principal | Connexion persistante | Reconnexion native navigateur | Cas idéal |
|---|---:|---:|---:|---|
| Polling | client → serveur | Non | Non nécessaire | Simplicité maximale |
| Long polling | client → serveur, réponse retardée | Semi-persistante | À coder | Compatibilité ancienne |
| SSE | serveur → client | Oui | Oui | Notifications, dashboards, flux texte |
| WebSocket | bidirectionnel | Oui | À coder | Chat, jeux, collaboration temps réel |

### 8.1 SSE vs polling

Polling :

```txt
Le client demande régulièrement : “du nouveau ?”
```

SSE :

```txt
Le serveur prévient quand il y a du nouveau.
```

SSE réduit donc les requêtes inutiles.

### 8.2 SSE vs WebSocket

WebSocket est bidirectionnel :

```txt
client ↔ serveur
```

SSE est unidirectionnel :

```txt
serveur → client
```

Mais cette limite est souvent une force :

```txt
Moins de possibilités = moins de complexité.
```

Pour une interface qui a surtout besoin de recevoir des avertissements, SSE est souvent plus simple que WebSocket.

## 9. Limites et pièges de SSE

### 9.1 SSE transporte du texte, pas du binaire

SSE envoie du texte UTF-8.

Pour envoyer des objets, on utilise généralement JSON :

```txt
data: {"type":"order_updated","id":123}

```

Pour du binaire lourd, SSE n’est pas adapté.

### 9.2 EventSource utilise principalement GET

L’API native `EventSource` ouvre une connexion avec `GET`.

Donc on n’envoie pas une commande critique avec SSE.

On fait plutôt :

```txt
client → serveur : POST /orders
serveur → client : SSE notification
client → serveur : GET /orders/{id} ou GET /sync
```

### 9.3 Les headers personnalisés sont limités

L’API native `EventSource` ne permet pas de définir librement des headers personnalisés comme avec `fetch`.

Pour l’authentification, on utilise souvent :

- cookies de session ;
- paramètres de requête courts et prudents ;
- reverse-proxy qui ajoute le contexte ;
- ou une librairie/polyfill si l’on veut plus de contrôle.

### 9.4 Les proxies peuvent bufferiser

Certains reverse-proxies tamponnent les réponses HTTP avant de les envoyer au client.

Pour SSE, c’est mauvais : les événements doivent arriver immédiatement.

Avec Nginx, on configure souvent :

```nginx
proxy_buffering off;
```

Et côté réponse :

```http
Cache-Control: no-cache
X-Accel-Buffering: no
```

### 9.5 Les connexions silencieuses peuvent être fermées

Il faut envoyer un heartbeat :

```txt
: heartbeat

```

Toutes les 15 à 30 secondes.

### 9.6 Attention aux workers multiples

Si FastAPI tourne avec plusieurs workers, une connexion SSE peut être ouverte dans un worker, alors qu’un événement est produit dans un autre.

Pour diffuser correctement les événements, il faut alors un mécanisme partagé :

- Redis Pub/Sub ;
- PostgreSQL `LISTEN/NOTIFY` ;
- broker de messages ;
- ou un seul worker au début.

## 10. Exemple minimal côté serveur avec FastAPI

Exemple pédagogique :

```python
import asyncio
import json
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse

app = FastAPI()


def sse_event(*, event: str, data: dict, event_id: int | None = None) -> str:
    lines = []

    if event_id is not None:
        lines.append(f"id: {event_id}")

    lines.append(f"event: {event}")
    lines.append(f"data: {json.dumps(data)}")

    return "\n".join(lines) + "\n\n"


@app.get("/stream")
async def stream(request: Request):
    async def event_generator():
        event_id = 0

        while True:
            if await request.is_disconnected():
                break

            event_id += 1
            yield sse_event(
                event="tick",
                event_id=event_id,
                data={"count": event_id},
            )

            await asyncio.sleep(1)

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )
```

## 11. Exemple minimal côté navigateur

```html
<script>
  const source = new EventSource("/stream");

  source.addEventListener("tick", (event) => {
    const payload = JSON.parse(event.data);
    console.log("tick", payload.count);
  });

  source.onerror = () => {
    console.log("Connexion SSE interrompue ou en reconnexion...");
  };
</script>
```

## 12. Tester SSE avec curl

`curl` peut afficher le flux SSE.

```bash
curl -N http://localhost:8000/stream
```

L’option `-N` demande à curl de ne pas bufferiser la sortie.

Résultat attendu :

```txt
id: 1
event: tick
data: {"count": 1}

id: 2
event: tick
data: {"count": 2}

```

## 13. Modèle mental à retenir

```txt
SSE = HTTP GET + réponse qui reste ouverte + messages texte structurés.
```

Et :

```txt
SSE ne remplace pas HTTP.
SSE complète HTTP.
```

Une bonne architecture ressemble souvent à ceci :

```txt
HTTP POST/PUT/DELETE = actions
HTTP GET             = lecture
SSE                  = notification serveur → client
/sync                = rattrapage robuste
```

---

## 14. Checklist technique SSE

Avant d’utiliser SSE en production, vérifier :

- [ ] `Content-Type: text/event-stream`
- [ ] `Cache-Control: no-cache`
- [ ] messages terminés par une ligne vide
- [ ] `id:` présent si on veut une reprise fiable
- [ ] heartbeat toutes les 15 à 30 secondes
- [ ] gestion de `Last-Event-ID`
- [ ] endpoint `/sync` si les événements doivent être rejoués
- [ ] désactivation du buffering reverse-proxy
- [ ] stratégie si le client est trop ancien ou trop en retard
- [ ] surveillance du nombre de connexions ouvertes
- [ ] stratégie multi-workers si plusieurs processus serveur

# Aller plus loin

## 1. Étudier HTTP/2

SSE est encore plus intéressant quand on comprend HTTP/2 :

- multiplexage ;
- streams ;
- une connexion TCP pour plusieurs flux ;
- réduction des limites HTTP/1.1.

Sujet à étudier :

```txt
Pourquoi HTTP/2 est meilleur pour plusieurs connexions longues ?
```

## 2. Étudier les reverse-proxies

SSE en production oblige à comprendre :

- Nginx ;
- Caddy ;
- Traefik ;
- buffering ;
- timeouts ;
- keep-alive.

Sujet à étudier :

```txt
Pourquoi proxy_buffering off est important pour SSE ?
```

## 3. Étudier les systèmes de diffusion

Dès que le serveur a plusieurs workers, il faut distribuer les événements :

- Redis Pub/Sub ;
- PostgreSQL LISTEN/NOTIFY ;
- RabbitMQ ;
- NATS ;
- Kafka pour des systèmes beaucoup plus grands.

Sujet à étudier :

```txt
Comment transmettre un événement à tous les workers FastAPI ?
```

## 4. Étudier les architectures event-driven

SSE devient très puissant quand l’application possède un journal d’événements :

```txt
event_log(id, type, entity_id, payload, created_at)
```

Sujet à étudier :

```txt
Quelle différence entre “état courant” et “journal d’événements” ?
```

## 5. Comparer SSE avec WebSocket sur un vrai prototype

Exercice conseillé :

1. faire un écran compteur avec polling ;
2. refaire le même avec SSE ;
3. refaire le même avec WebSocket ;
4. couper le Wi-Fi ;
5. observer ce qui se passe.

Le vrai apprentissage arrive souvent avec les pannes.
