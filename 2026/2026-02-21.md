# Linux & Docker — Synthèse

> Concepts appris et compris, avec exemples concrets

---

## 1. Le noyau Linux

Le noyau (kernel) est le **cœur du système d'exploitation** — un binaire C compilé qui tourne en mémoire dès le démarrage. Il fait l'intermédiaire entre le matériel et les programmes.

```bash
# Voir ton noyau
uname -r                          # 6.8.0-100-generic
ls -lh /boot/vmlinuz-$(uname -r)  # le binaire, ~15 Mo compressé
sudo file /boot/vmlinuz-$(uname -r)
# → Linux kernel x86 boot executable bzImage
```

**Ce que fait le noyau :**
- Gère les processus (scheduler, PID)
- Gère la mémoire (RAM, swap)
- Gère les fichiers (ext4, /proc, /dev)
- Gère le matériel via les drivers
- Gère le réseau (pile TCP/IP)
- Isole les processus via namespaces et cgroups

**La frontière fondamentale :**
```
User space   → tes programmes (bash, Docker, FastAPI...)
─────────────────────────────── syscall interface
Kernel space → le noyau (accès direct au matériel)
```

## 2. Les syscalls

Un **syscall** (system call) est la façon dont un programme demande au noyau de faire quelque chose de bas niveau. Les programmes ne peuvent pas accéder directement au matériel.

```bash
# Espionner les syscalls d'un programme
strace ls
# tu verras : openat(), read(), write()...
```

**Syscalls courants :**
```
open()   → ouvrir un fichier
read()   → lire des données
write()  → écrire des données
fork()   → créer un processus enfant
exec()   → charger un programme
socket() → créer une connexion réseau
exit()   → terminer un processus
```

## 3. Systemd

Systemd est le **premier processus lancé par le noyau** (PID 1). Il orchestre le démarrage de tous les services.

```bash
# Voir systemd en PID 1
ps aux | head -5
# root  1  /sbin/init splash  ← c'est systemd

# /sbin/init est un lien symbolique vers systemd
ls -la /sbin/init

# Gérer les services
systemctl start docker
systemctl enable docker   # démarre au boot
systemctl status docker

# Voir l'ordre de démarrage
systemd-analyze blame | head -20
```

**Systemd vs systemctl :**
```
systemd    → le programme (PID 1, fait le travail)
systemctl  → l'outil pour lui donner des ordres
```

## 4. Les processus et le fork

Tout programme qui tourne est un **processus** avec un PID unique.

```bash
echo $$           # PID de ton bash
ps aux            # tous les processus
pstree -p         # arbre des processus
ps -ef | grep nom # voir le PPID (parent)
```

**Le fork :** quand bash lance un programme, il se **duplique** (fork). Le duplicata devient le nouveau programme. Bash attend que l'enfant termine, puis reprend la main.

```bash
sleep 30 &           # lance sleep en arrière-plan
ps -ef | grep sleep  # son PPID = ton bash
```

## 5. Tout est fichier

En Linux, **tout est représenté comme un fichier** — y compris les processus, les devices, les sockets.

```bash
# /proc — fenêtre sur le noyau (fichiers virtuels, générés en temps réel)
cat /proc/version          # infos sur le noyau
cat /proc/cpuinfo          # infos sur le CPU
cat /proc/meminfo          # infos sur la RAM
cat /proc/$$/status        # état de ton bash
ls /proc/$$/fd             # file descriptors ouverts : 0, 1, 2...
```

**Types de fichiers :**
```
-   fichier ordinaire
d   directory (dossier)
l   lien symbolique
b   block device (disque)
c   character device (clavier, terminal)
p   pipe
s   socket
```

## 6. Les file descriptors

Chaque processus a un **tableau numéroté** de connexions ouvertes.

```
0 → stdin  (entrée — clavier par défaut)
1 → stdout (sortie — écran par défaut)
2 → stderr (erreurs — écran par défaut)
```

**Les redirections et pipes :**
```bash
commande > fichier.txt   # fd 1 → fichier
commande 2> erreurs.txt  # fd 2 → fichier
commande < input.txt     # fd 0 ← fichier
cmd1 | cmd2              # fd 1 de cmd1 → fd 0 de cmd2
```

Le pipe connecte toujours **stdout du gauche → stdin du droit**.

## 7. Les namespaces

Les namespaces sont des **vues isolées de ressources du noyau**. C'est la technologie fondamentale derrière Docker.

```bash
lsns   # voir tous les namespaces actifs sur ta machine
```

**Types de namespaces :**
```
PID   → liste des processus (le conteneur croit être seul)
NET   → interfaces réseau et ports
MNT   → système de fichiers
UTS   → hostname
USER  → utilisateurs et permissions
IPC   → communication entre processus
```

**Chrome et VS Code utilisent déjà des namespaces** sur ta machine pour isoler leurs processus internes. Docker fait la même chose, mais pour des apps entières avec leur propre système de fichiers.

```bash
# Créer un namespace PID sans Docker
sudo unshare --pid --fork bash
echo $$   # tu es PID 1 dans ce namespace !
exit
```

## 8. Docker — les concepts fondamentaux

### Image
Un **système de fichiers figé en couches** — une recette reproductible. Ne tourne pas, ne consomme pas de CPU.

### Conteneur
Une **instance vivante** d'une image. Un processus isolé via namespaces qui croit être dans son propre système. Partage le noyau de l'hôte.

### La différence avec une VM
```
VM         → duplique le noyau, lourd (Go, minutes à démarrer)
Conteneur  → partage le noyau, léger (Mo, secondes à démarrer)
```

### Le conteneur remplace Ubuntu, garde le noyau
```
Ton Ubuntu
├── Noyau Linux 6.8      ← PARTAGÉ
└── systemd, GNOME...    ← le conteneur ne voit rien de ça

Conteneur
├── Noyau Linux 6.8      ← le même, emprunté
├── mini Debian slim      ← son propre système
└── ton app              ← isolée
```

## 9. Le Dockerfile

Le Dockerfile est la **recette pour construire une image**. Il vit dans GitHub avec ton code.

```dockerfile
FROM python:3.12-slim    # image de base (Debian + Python préinstallé)
WORKDIR /app             # dossier de travail dans le conteneur
COPY . .                 # copie tout ton code dedans
CMD ["python", "main.py"] # commande au démarrage
```

**Les instructions clés :**
```
FROM     → point de départ (image de base)
COPY     → copier des fichiers dans l'image
RUN      → exécuter une commande pendant le build
WORKDIR  → définir le dossier courant dans le conteneur
CMD      → commande lancée au démarrage du conteneur
ENV      → variable d'environnement
EXPOSE   → port exposé (documentation)
```

## 10. Les commandes Docker essentielles

```bash
# Construire une image
docker build -t mon-app .

# Lancer un conteneur
docker run mon-app
docker run -e NOM="Paul" mon-app    # avec variable d'environnement
docker run -d mon-app               # en arrière-plan (detached)
docker run -p 8000:8000 mon-app     # exposer un port

# Voir les images
docker images

# Voir les conteneurs
docker ps        # conteneurs en cours
docker ps -a     # tous les conteneurs (y compris arrêtés)

# Logs
docker logs mon-conteneur

# Supprimer
docker rm mon-conteneur   # supprimer un conteneur
docker rmi mon-app        # supprimer une image
```

## 11. Le workflow complet

```
# Développement
git clone mon-projet
docker build -t mon-app .
docker run mon-app           # tester en local

# Livraison
docker build -t ghcr.io/paul/pos-api:v1.0 .
docker push ghcr.io/paul/pos-api:v1.0

# Déploiement Cotonou
docker pull ghcr.io/paul/pos-api:v1.0
docker compose up

# Ou via clé USB
docker save pos-api:latest > pos-api.tar   # exporter
docker load < pos-api.tar                  # importer
```

## 12. Choix des images de base

```
python:3.12-slim    → apps Python (FastAPI) — Debian allégé, stable
postgres:16-alpine  → PostgreSQL — Alpine, très léger, bien packagé
nginx:alpine        → Nginx — Alpine, minimal
ubuntu:22.04        → si besoin d'un système complet
```

**Règle générale :**
- `slim` pour les apps Python/Node (compatibilité glibc)
- `alpine` pour les services stables (PostgreSQL, Nginx, Redis)

## 13. Application à ton POS

```yaml
# docker-compose.yml
services:
  api:
    build: .                          # construit depuis ton Dockerfile
    image: ghcr.io/paul/pos-api:v1.0
    environment:
      - DATABASE_URL=postgresql://...
    ports:
      - "8000:8000"
    depends_on:
      - db

  db:
    image: postgres:16-alpine         # image officielle PostgreSQL
    environment:
      - POSTGRES_PASSWORD=secret
    volumes:
      - postgres_data:/var/lib/postgresql/data  # données persistantes

volumes:
  postgres_data:
```

```bash
# Sur le serveur à Cotonou
docker compose up -d    # tout démarre en une commande
```
