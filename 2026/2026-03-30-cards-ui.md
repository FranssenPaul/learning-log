# Learning Log — Comprendre les cards UI

## 1. L’idée générale

Une **card UI** est un bloc d’interface qui regroupe un contenu, une information ou une action autour d’un même sujet.

On l’utilise très souvent pour :

- présenter une information
- isoler une section
- structurer une page
- mettre en avant une action
- rendre une interface plus lisible

Exemples très courants :

- une card de profil
- une card produit
- une card de statistiques
- une card de message
- une card d’action

L’idée importante est la suivante :

> une card aide à rassembler ce qui va ensemble, pour que l’utilisateur comprenne plus vite ce qu’il regarde et ce qu’il peut faire.

---

## 2. Qu’est-ce qu’une card, simplement ?

Une card est un **conteneur visuel**.

Elle sert à donner une forme claire à un petit ensemble cohérent.

On peut la voir comme une sorte de “boîte d’interface” qui contient par exemple :

- un contexte
- un titre
- un texte
- un contenu utile
- une ou plusieurs actions

Une card n’est pas obligée d’être compliquée.

Elle peut être :

- très simple
- très riche
- purement informative
- surtout orientée action

---

## 3. Pourquoi les cards sont utiles ?

Les cards sont utiles parce qu’elles apportent plusieurs choses en même temps :

- elles **séparent visuellement** les informations
- elles **créent une hiérarchie** plus claire
- elles **rendent une interface plus facile à scanner**
- elles **encadrent mieux les actions**

Sans cards, une page peut vite devenir une suite de blocs difficiles à distinguer.

Avec des cards, on comprend mieux :

- où commence une information
- où elle se termine
- quelle est son importance
- quelle action lui est liée

---

## 4. Anatomie simple d’une card

Toutes les cards ne possèdent pas les mêmes parties, mais on retrouve souvent les zones suivantes :

1. le conteneur principal
2. une zone haute de contexte
3. un titre
4. un texte explicatif
5. une zone de contenu
6. une zone d’actions
7. parfois une zone basse séparée

En anglais, on rencontre souvent ces mots :

- `Card`
- `Header`
- `Title`
- `Description`
- `Content` ou `Body`
- `Actions`
- `Footer`

Vue très simple :

```text
+--------------------------------------------------+
| Header                                           |
| Title                                            |
| Description                                      |
|                                                  |
| Content / Body                                   |
|                                                  |
| Actions                                          |
+--------------------------------------------------+
```

## 5. Vocabulaire essentiel FR / EN

Voici le vocabulaire le plus utile à connaître.

### `Card` → carte / bloc / conteneur de contenu

Le mot `card` désigne le composant complet.

C’est le bloc global qui rassemble les éléments d’un même sujet.

### `Header` → en-tête

Le `header` est la zone supérieure de la card.

Il contient souvent :

- le contexte
- le titre
- une icône
- un badge
- parfois une action

### `Title` → titre

Le `title` est le texte principal.

C’est ce qui nomme clairement la card.

Exemples :

- `Profil`
- `Dernière commande`
- `Paramètres`

### `Eyebrow` → sur-titre / petit texte de contexte

Le mot `eyebrow` est très utilisé en design UI.

Il désigne le petit texte placé au-dessus du titre principal.

Son rôle :

- introduire le contexte
- indiquer une catégorie
- donner un niveau de lecture plus discret

Visuellement, il est souvent :

- plus petit
- moins fort que le titre
- parfois en capitales

### `Subtitle` → sous-titre

Le `subtitle` est un texte secondaire placé sous le titre ou à côté de lui.

Il complète le titre sans le remplacer.

### `Description` → description / texte explicatif

La `description` sert à expliquer :

- ce que contient la card
- ce que l’utilisateur peut faire
- ce qu’il faut comprendre

Elle est plus développée que le titre.

### `Body` ou `Content` → corps / contenu

Le `body` ou `content` est la zone utile principale.

C’est l’endroit où l’on trouve par exemple :

- du texte
- un champ
- une liste
- une image
- un tableau
- un graphique

Dans beaucoup d’équipes :

- `body` insiste sur la structure
- `content` insiste sur le contenu lui-même

En pratique, les deux sont très proches.

### `Footer` → pied de card / zone basse

Le `footer` est la zone située en bas de la card.

On y place souvent :

- des boutons
- des liens
- des informations secondaires

Toutes les cards n’ont pas de footer.

### `Actions` → actions

Les `actions` sont les éléments qui permettent d’agir.

Exemples :

- `Voir plus`
- `Modifier`
- `Supprimer`
- `Continuer`

### `Primary action` → action principale

C’est l’action la plus importante de la card.

Celle que l’utilisateur est le plus susceptible d’utiliser.

### `Secondary action` → action secondaire

C’est une action utile, mais moins prioritaire.

Elle est souvent plus discrète.

### `Badge` → badge / pastille d’état

Un `badge` sert à montrer un état ou une catégorie.

Exemples :

- `Nouveau`
- `En ligne`
- `Brouillon`
- `Urgent`

### `Meta` ou `Metadata` → métadonnées / informations secondaires

La `meta` regroupe les informations secondaires.

Exemples :

- une date
- un auteur
- un statut
- un nombre d’éléments

### `Media` → média visuel

Le `media` correspond à un contenu visuel intégré à la card :

- image
- miniature
- illustration
- vidéo

### `Variant` → variante

Une `variant` est une variation visuelle ou fonctionnelle d’un même composant.

Exemples :

- card sobre
- card avec bordure
- card avec ombre
- card interactive

## 6. Toutes les cards n’ont pas la même structure

Une erreur fréquente est de croire qu’une card doit toujours contenir tous les éléments possibles.

En réalité, une card peut être :

- minimale
- informative
- interactive
- visuelle
- utilitaire

### Card minimale

```text
Title
Action
```

### Card informative

```text
Eyebrow
Title
Description
```

### Card de contenu

```text
Header
Content
Footer
```

### Card orientée action

```text
Eyebrow
Title
Action principale
```

Ce qu’il faut retenir :

> une bonne card n’est pas une card qui contient “tout”, c’est une card dont les éléments sont utiles et bien hiérarchisés.

---

## 7. Rôle de chaque élément dans une card

Pour construire une card claire, il faut comprendre le rôle de chaque partie.

- **Eyebrow** : donne le contexte rapidement
- **Title** : nomme clairement le sujet
- **Description** : explique ce qu’il faut comprendre
- **Content / Body** : porte l’information ou l’interaction principale
- **Actions** : permettent d’agir
- **Footer** : regroupe les éléments secondaires du bas
- **Badge / Meta** : ajoutent des informations complémentaires

Quand cette hiérarchie est bonne, l’utilisateur lit la card plus naturellement.

## 8. Exemple simple et lisible en HTML / CSS

Voici un exemple volontairement simple, pour bien voir les parties principales d’une card.

```html
<article class="card">
  <div class="card-header">
    <p class="card-eyebrow">Compte</p>
    <h2 class="card-title">Profil utilisateur</h2>
    <p class="card-description">
      Retrouvez vos informations principales et accédez aux réglages.
    </p>
  </div>

  <div class="card-content">
    <p><strong>Nom :</strong> Alice Martin</p>
    <p><strong>Statut :</strong> Active</p>
  </div>

  <div class="card-footer">
    <button class="primary-action">Voir le profil</button>
    <button class="secondary-action">Modifier</button>
  </div>
</article>
```

## 9. Comment lire cette card

Dans cet exemple :

- `article.card` = la **card**
- `div.card-header` = le **header**
- `p.card-eyebrow` = l’**eyebrow**
- `h2.card-title` = le **title**
- `p.card-description` = la **description**
- `div.card-content` = le **content / body**
- `div.card-footer` = le **footer**
- les boutons = les **actions**

Ce petit exemple permet déjà de comprendre l’essentiel :

- une card est un conteneur
- elle peut être structurée par zones
- chaque zone a un rôle

## 10. Principes simples pour faire une bonne card

### 1. Avoir un but clair

Une card doit servir à quelque chose de précis.

Par exemple :

- informer
- faire choisir
- résumer
- lancer une action

Si la card fait trop de choses à la fois, elle devient confuse.

### 2. Montrer une hiérarchie visuelle claire

L’utilisateur doit comprendre rapidement :

- ce qui est le contexte
- ce qui est le titre
- ce qui est important
- ce qui est secondaire

### 3. Ne pas surcharger

Une card n’a pas besoin d’accueillir trop de texte, trop de boutons ou trop de niveaux visuels.

Trop d’éléments nuisent à la lisibilité.

### 4. Rendre l’action évidente

S’il existe une action principale, elle doit ressortir visuellement.

### 5. Garder de la cohérence

Si une interface utilise plusieurs cards, elles doivent partager une logique commune :

- espacements
- titres
- placements des actions
- style général

Cette cohérence aide beaucoup à la compréhension.

## 11. Lien avec les outils UI modernes

Dans les outils UI actuels, les cards sont partout.

On les retrouve dans :

- les design systems
- les bibliothèques de composants
- les kits d’interface
- les frameworks frontend

Voici quelques familles d’outils très courantes aujourd’hui.

### 1. Bibliothèques de composants complètes

Exemples :

- **Material UI (MUI)**
- **Chakra UI**
- **Ant Design**
- **Bootstrap**

Leur idée :

- proposer des composants déjà prêts
- avec des conventions déjà définies
- et souvent une card déjà fournie

#### Avantages

- démarrage rapide
- beaucoup de composants disponibles
- documentation souvent riche
- cohérence immédiate

#### Inconvénients

- style parfois reconnaissable ou standardisé
- personnalisation parfois plus lourde
- on peut se sentir “dans le cadre” de la bibliothèque

### 2. Approches plus ouvertes et composables

Exemples :

- **shadcn/ui**
- **Radix UI** pour les primitives

Leur idée :

- fournir une base de composants ou de primitives
- laisser davantage de contrôle sur le rendu final
- faciliter la construction d’un design system plus personnel

#### Avantages

- plus de liberté de personnalisation
- meilleure maîtrise du code final
- très adapté aux interfaces sur mesure

#### Inconvénients

- demande plus de décisions de design
- demande souvent plus de travail au départ
- moins “prêt à poser” qu’une bibliothèque complète

### 3. Approches utilitaires centrées CSS

Exemples :

- **Tailwind CSS**
- **Tailwind UI / Tailwind Plus** pour des blocs prêts à l’emploi

Ici, l’idée est souvent :

- construire la card avec des classes utilitaires
- ou partir de blocs déjà conçus

#### Avantages

- grande rapidité pour prototyper
- forte liberté visuelle
- très bon contrôle fin du rendu

#### Inconvénients

- le HTML peut devenir chargé
- il faut garder une discipline de structure
- sans méthode, le design peut devenir incohérent

## 12. Comment choisir un outil UI pour faire des cards ?

Le bon choix dépend surtout du besoin.

### Si tu veux aller vite

Une bibliothèque comme **MUI**, **Chakra UI** ou **Bootstrap** peut être rassurante.

Tu obtiens rapidement :

- une card existante
- une structure connue
- des variantes prêtes

### Si tu veux une interface plus personnalisée

Une approche comme **shadcn/ui** ou **Radix UI** est souvent plus souple.

Tu gardes plus de contrôle sur :

- la structure
- le style
- le comportement

### Si tu veux apprendre la structure en profondeur

Faire quelques cards en **HTML/CSS simple** ou avec **Tailwind CSS** est très formateur.

Tu comprends mieux :

- les zones
- la hiérarchie
- les espacements
- le rôle réel de chaque élément

## 13. Avantages et limites des cards elles-mêmes

Les cards sont très utiles, mais elles ne résolvent pas tout.

### Avantages

- lisibilité
- modularité
- bonne séparation visuelle
- réutilisation facile
- très utiles dans les dashboards, profils, listes et pages d’accueil

### Inconvénients

- trop de cards peuvent fragmenter la lecture
- une mauvaise hiérarchie rend la card confuse
- certaines interfaces n’ont pas besoin de cards partout
- une card peut devenir un “sac à tout” si on y met trop d’éléments

Il faut donc utiliser les cards avec intention, pas simplement parce qu’elles sont à la mode.

## 15. À explorer ensuite

- [ ] **La hiérarchie visuelle**
  Comprendre la taille du texte, les espacements, le contraste et l’ordre de lecture aide énormément à concevoir de bonnes cards.

- [ ] **Les états d’une card**
  Une card peut avoir des états : normale, survolée, sélectionnée, désactivée, chargement. C’est une bonne suite logique après l’anatomie.

- [ ] **Les cards interactives**
  Certaines cards sont entièrement cliquables, d’autres contiennent plusieurs boutons. Cela pose des questions de clarté et d’accessibilité.

- [ ] **L’accessibilité**
  Il est utile de comprendre comment structurer une card pour qu’elle reste claire avec le clavier, les lecteurs d’écran et une bonne sémantique HTML.

- [ ] **Les design systems**
  Une card prend vraiment tout son sens quand elle s’inscrit dans un système plus large de composants cohérents.

- [ ] **Les variantes**
  Comprendre les variantes aide à passer d’une seule card à une famille de cards cohérentes.

- [ ] **Le responsive**
  Une card peut très bien fonctionner sur desktop et beaucoup moins bien sur mobile. Observer ce point est très formateur.
