# React — Learning Log
### À la sauce Hofstadter : de l'analogie à la cristallisation

---

## Point de départ : la SPA

Tout part d'une observation simple : une application React est une **Single Page Application**. Le serveur envoie `index.html` **une seule fois**. Ensuite, plus aucune requête pour naviguer — tout se passe à l'intérieur, dans le navigateur.

C'est le fondement. Tout le reste s'y accroche.

---

## Le composant est une fonction

Pas une métaphore — une réalité littérale.

```jsx
const Bouton = ({ label, onClick }) => {
  return <button onClick={onClick}>{label}</button>
}
```

- Le composant **est** une fonction
- Les **props** sont ses paramètres
- Le JSX retourné est une **description** de ce qui doit s'afficher — pas une action, pas une modification

> Vite traduit ce JSX en JavaScript pur que le navigateur comprend. Le bundle final — JS, CSS, HTML — est déposé une fois sur le serveur. C'est tout.

---

## Le cycle React — la mécanique centrale

```
état change (setCount, dispatch...)
    → React réexécute la fonction composant
    → produit un nouveau Virtual DOM
    → diff avec l'ancien Virtual DOM   ← réconciliation
    → patch minimal sur le vrai DOM
```

### Le Virtual DOM

React maintient **deux** représentations légères en mémoire — des objets JavaScript ordinaires qui décrivent ce que le DOM devrait être :

- **L'ancien** — correspond au DOM réel actuellement affiché
- **Le nouveau** — produit par le dernier re-render

La **réconciliation** compare les deux, calcule le diff, et n'applique au vrai DOM que ce qui a changé. Comme un correcteur qui ne réécrit pas la page entière — il cherche les différences et ne corrige que ce qui a changé.

> Le composant **décrit**. React **décide** comment appliquer. C'est le contrat déclaratif.

## Les Hooks — un système

Avant les hooks, distinction clé :

- **Composant** : la fonction (le "plan"), par exemple `TodoItem`.
- **Instance de composant** : une occurrence réelle à l'écran (une "copie montée") de ce plan.

Si tu affiches 3 fois `<TodoItem />`, tu as **1 composant** (la fonction) mais **3 instances** (3 mémoires séparées).

Un hook est une **fonction côté code** (`useState`, `useEffect`, `useRef`...).  
Mais son appel pointe vers une **cellule interne côté React**.

Concrètement, il y a 2 niveaux :

- **Dans ton code** : tu appelles une fonction JavaScript.
- **Dans React** : React lit/écrit une case de données attachée à la **mémoire interne de cette instance**.

Donc un hook n'est pas "juste une fonction" ni "juste un objet" : c'est une fonction qui donne accès à un stockage interne géré par React.

Cette donnée est **persistante entre les renders** tant que l'instance reste montée.  
Si le composant est démonté puis remonté (ou remount via `key`), cette mémoire repart de zéro.

> C'est pour ça qu'un hook ne peut jamais être dans un `if` : React associe les cellules par **ordre d'appel**. Si l'ordre change, il lit la mauvaise case.

<sub>Note avancée : dans la doc interne React, cette mémoire d'instance est portée par une structure appelée *fiber*.</sub>

### useState — l'état du bâtiment

```jsx
const [statut, setStatut] = useState('libre')
```

Comme **l'état d'un bâtiment** qui se dégrade et qu'on remet à neuf. Quand l'état change, React déclenche une rénovation — un re-render complet.

`setStatut('occupée')` ne modifie pas une variable. Il dit à React :
> "Photographie ce composant avec cette nouvelle valeur."

La fonction est réexécutée depuis le début avec le nouvel état. `statut` était une constante dans l'ancien render — elle reste une constante dans le nouveau, mais avec une valeur différente.

**Machine à état basique** : 1 valeur, 1 transition pour la changer.

---

### useEffect — le scorpion

```jsx
useEffect(() => {
  // s'exécute après le render
}, [dépendance])
```

Le scorpion **attend que tout soit calme** — après que le render et la réconciliation sont terminés, après que le vrai DOM est à jour. Puis il agit : requête API, abonnement, timer, manipulation directe du DOM.

Ses **pinces** — les crochets — contrôlent précisément quand il frappe :

```jsx
useEffect(() => { ... })          // frappe après chaque render
useEffect(() => { ... }, [])      // frappe une seule fois au montage
useEffect(() => { ... }, [count]) // frappe quand count change
```

Après chaque render, React compare les dépendances avec celles du render précédent. Si une valeur a changé — le scorpion frappe. Sinon — il attend.

> C'est encore du diff, comme la réconciliation. Même invariant, appliqué ailleurs.

---

### useRef — le gabarit de niveau

```jsx
const inputRef = useRef(null)
inputRef.current.focus()
```

Comme les **planches de gabarit** qu'on plante sur un chantier pour fixer le niveau — posées une fois, stables pendant toute la durée des travaux, consultées quand on en a besoin. Elles ne déclenchent aucun travaux, elles servent de **référence fixe**.

useRef pointe vers un même objet à travers tous les renders. Modifier `ref.current` ne déclenche **aucun re-render**. C'est une persistance **inconditionnelle et silencieuse**.

---

### useMemo — la note de calcul

```jsx
const total = useMemo(() => {
  return commandes.reduce((sum, c) => sum + c.montant, 0)
}, [commandes])
```

Comme une **note de calcul de structure** — elle reste valable tant que les charges et les matériaux n'ont pas changé. Si rien ne bouge, inutile de la refaire. Dès qu'une dépendance change — on recalcule.

Comme useRef, useMemo pointe vers un résultat stable à travers les renders. Mais c'est une persistance **conditionnelle** — elle tient jusqu'au prochain changement de dépendance.

```
useRef    → même objet        — persistance inconditionnelle
useMemo   → même résultat     — persistance conditionnelle
useState  → nouvel état       — persistance avec re-render
```

### useReducer — le système de transitions

```jsx
const [state, dispatch] = useReducer(reducer, initialState)
dispatch({ type: 'AJOUTER_COMMANDE', payload: commande })
```

useState est une **machine à état basique** — une valeur, une transition pour la changer.

useReducer est une **machine à état complète** — un état complexe, un jeu de transitions définies à l'avance, chacune nommée et explicite.

```js
const reducer = (state, action) => {
  switch(action.type) {
    case 'OUVRIR_TABLE':
      return { ...state, statut: 'occupée' }
    case 'AJOUTER_COMMANDE':
      return { ...state, commandes: [...state.commandes, action.payload] }
    case 'FERMER_TABLE':
      return { ...state, statut: 'libre', commandes: [] }
    default:
      return state
  }
}
```

Ici, `...` est l'opérateur **spread** :

- `...state` copie les propriétés de l'ancien objet dans un **nouvel objet**.
- Puis `statut: 'occupée'` remplace uniquement cette propriété.

Donc `return { ...state, statut: 'occupée' }` veut dire :  
"garde tout l'état tel quel, sauf `statut`."

Même idée pour le tableau :  
`[...state.commandes, action.payload]` = "reprend toutes les commandes, puis ajoute la nouvelle à la fin".

On fait ça pour rester **immutable** : on ne modifie pas l'ancien état, on crée une nouvelle version.

Le reducer est défini **en amont** — comme un plan de transitions. Chaque `dispatch` déclenche la transition correspondante, qui produit le nouvel état, qui provoque le re-render.

> useState est un useReducer simplifié où le reducer est trivial — il reçoit directement la nouvelle valeur. useReducer est le représentant canonique. useState est un cas particulier.

## Le cycle complet

```
état change (useState ou dispatch)
    → React réexécute la fonction composant
    → nouveau Virtual DOM produit
    → réconciliation avec l'ancien
    → patch minimal sur le vrai DOM
    → useEffect réagit si ses dépendances ont changé
    → éventuellement nouvel état → nouveau cycle
```

---

## Ce que tout cela signifie

React est un **paradigme déclaratif et fonctionnel** appliqué aux interfaces. Ses principes fondateurs sont cohérents entre eux :

- L'UI est une **fonction de l'état**
- Les composants sont des **fonctions pures**
- Les effets de bord sont **isolés** dans useEffect
- La donnée **descend** par les props
- Les événements **remontent** par les callbacks

Tout découle de ces principes. Ce n'est pas une collection de recettes — c'est une vision cohérente.

> Le moteur React est dans le bundle — il voyage avec l'application. `react` gère la logique pure, `react-dom` fait le pont vers le vrai DOM. C'est pour ça que React Native existe — même moteur, pont différent vers les composants mobiles natifs.
